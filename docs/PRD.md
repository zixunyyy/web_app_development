# 任務管理系統 PRD (Product Requirements Document)

## 1. 專案概述
- **背景與動機**：在快節奏的工作環境中，個人與團隊常面臨任務繁雜、進度難以追蹤、甚至遺漏重要事項的痛點。本系統旨在提供一個輕量、直覺的管理工具，幫助使用者有效組織工作項目，並確保每個任務按時完成。
- **目標用戶**：需要管理日常任務的個人、專案經理以及小型工作團隊。
- **核心價值主張**：提供一個集中式、清晰易用的平台，協助使用者梳理任務優先級，即時掌握進度並降低協作溝通成本。

## 2. 功能需求

1. **任務 CRUD (建立/讀取/編輯/刪除)**
   - 作為 **使用者**，我希望 **能夠建立新的任務並隨時修改或刪除它們**，以便 **準確記錄我需要做的事情與變更**。
2. **任務狀態追蹤與切換**
   - 作為 **團隊成員**，我希望 **能夠將任務狀態切換為「待辦 (To Do)」、「進行中 (In Progress)」或「已完成 (Done)」**，以便 **反應任務當前的執行進度**。
3. **優先級與到期日設定**
   - 作為 **使用者**，我希望 **能為任務設定優先級別（高/中/低）與截止日期**，以便 **我了解每天該先處理哪些重要且緊急的事情**。
4. **任務分類與標籤**
   - 作為 **專案經理**，我希望 **能替任務分派標籤或歸類到特定專案下**，以便 **在大量專案中能快速過濾與搜尋特定任務**。
5. **儀表板觀測與提醒**
   - 作為 **使用者**，我希望 **能在登入後的首頁儀表板看見「即將到期」與「今日待辦」的任務清單**，以便 **快速展開每天的工作流程**。

## 3. 非功能需求
- **技術限制**：系統將使用 Flask（後端） + Jinja2 模板（前端） + SQLite（資料庫）開發。
- **效能考量**：系統回應時間應在 1-2 秒內；資料庫架構需能支援日常任務操作的快速查詢（例如使用 Index）。
- **安全考量**：基本的防護措施（例如避免 SQL Injection、XSS），如需多用戶支援則需實作密碼雜湊（Hashing）與 Session 驗證。

## 4. MVP (Minimum Viable Product) 範圍

- **Must Have（必須具備，首要上線目標）**：
  - 任務的建立、檢視、編輯、刪除 (CRUD)
  - 任務狀態變更與清單顯示
- **Should Have（應該具備，優化體驗）**：
  - 設定任務的優先順序與截止日期
  - 任務的分類（清單/標籤）
- **Nice to Have（最好具備，進階亮點）**：
  - 多使用者註冊與登入
  - 視覺化統計儀表板
